# social-retail 后端类图结构说明

> 按功能模块分类，分为**后端控制层**和**业务逻辑层**两大类，描述各类之间的依赖关系与职责。

---

## 一、项目分层架构总览

```
┌─────────────────────────────────────────────────────┐
│                 Controller Layer                     │
│  (admin / merchant / user / order / pay)            │
├─────────────────────────────────────────────────────┤
│                  Service Layer                       │
│  (admin / merchant / product / order / pay /        │
│   promotion / social / member / analysis /          │
│   address / sms / user)                             │
├─────────────────────────────────────────────────────┤
│                  Mapper Layer                        │
│  (address / member / order / product / user)        │
├─────────────────────────────────────────────────────┤
│                  Entity Layer                        │
│  (address / member / order / product / sms / user)  │
└─────────────────────────────────────────────────────┘
```

## 五、关键设计要点

### 5.1 分层架构规范

- **Controller 层**：仅负责参数接收、校验、调用 Service、返回封装结果（`Result<T>` / `PageResult<T>`）。使用 `@RestController` + `@RequestMapping` 注解。
- **Service 层**：承担全部业务逻辑，通过 `@Service` 注解声明，通过 `@Resource` 注入 Mapper。多数模块采用 Interface + Impl 模式。
- **Mapper 层**：数据访问，继承 MyBatis-Plus `BaseMapper<T>`。

### 5.2 特殊说明

- `MerchantProductService` 是**实体类而非接口**，直接在 `MerchantGoodsController` 中注入使用。
- `MerchantOrderController` 同时注入了 Mapper 和 Service，是项目中依赖最多、功能最复杂的 Controller。
- promotion / social / member / analysis 四个模块**无独立 Controller**，其能力被其他模块的 Controller 和定时任务调用。
- `AdminService` 接口暂无实现类。
- 基础设施层（`infrastructure/`）提供支付、短信、OSS 等第三方通道封装，供 Service 层调用。
